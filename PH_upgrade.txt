# 需要修改的部分
1. Plan-Exe部分
1.1 Planner应该知道如果它需要执行的内容目前的工具无法做到，就升级SKILL或创建新skill
1.2 Executor应该只知道当前步骤、上一步结果、以及所需要的输出以做上下文隔离
2. 工具部分
2.1 tools, mcp, SKILL应该明确能力边界，目前边界不明确，要明白分析方法有很多种，用户会提出详细的分析方法，即使是回归分析也会有很多种，但是不清楚的能力边界会导致使用了错误的工具
2.2 加入修改、查询和创建SKILL的能力，并将SKILL作为可以选择启用哪些的
2.3 为用户明确给出插入MCP的插口
3. 配置部分
3.1 应当允许deepseek, longcat等第三方模型的接入
3.2 为每一个用户的user_id保存其api_key。保存记忆也是通过user_id保存的，需要根据用户隔离。
4. 说明文档
4.1 README应当简要，最重要的是告诉用户如何快速的启动这个项目
5. 前端
5.1 前端之间各部分应该可以点击返回，避免只能通过重新输入URI进入
5.2 让用户可以在前端输入API_KEY等配置
5.3 提高前端美观度，例如动态效果等
6. 项目整体
6.1 给项目封装，提供一个简单的入口，使得其能够通过一个入口进入，而不需要前后端分别开启。
6.2 说明如何验证分析结果的正确性

# 注意
1. 新功能必须完整集成到项目中并实际可用，必须找到与其可能有关联的所有部分处理好依赖关系，与其他部分进行集成测试，并更新对应说明文档。
2. 成熟项目依旧是你应该优先参考的对象，优先级为：照搬成熟项目+调整到可用>依照项目自己编写>理解逻辑后自己编写。无论是提示词还是SKILL都可以优先去本地的：

# 增量测试原则

1. **测试顺序**：前序模块验证 → 新增模块测试 → 模块间集成测试
2. **测试覆盖**：每个模块必须有对应的单元测试
3. **集成测试**：模块间集成必须有集成测试
4. **测试调整**：测试逻辑可根据实际需求调整，但必须保证测试目标
5. **自动化**：所有测试应可自动化执行，确保持续集成
